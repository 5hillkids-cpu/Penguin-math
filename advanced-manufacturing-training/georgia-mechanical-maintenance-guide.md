# Georgia Mechanical Maintenance Guide

## Purpose
This guide defines how to maintain, extend, and version the Georgia Mechanical training package over time.

## File Naming Convention
- Use lowercase letters and hyphens only.
- Keep filenames descriptive and role-oriented.
- Prefix numbered module files when order matters.
- Use company-specific prefixes only when the file is a branded derivative.

## Current Naming Pattern
- Module content: numbered markdown files inside each module folder
- Program assets: georgia-mechanical-* files at the training root
- Print assets: HTML files inside print-templates/
- Handoff navigation: README.md and index.html files inside handoff-package/

## Versioning Guidance
- Add a short version note whenever a module changes substantially.
- Record date, editor, and summary of change.
- If a client-specific variant is created, keep the generic source file when possible.

## Suggested Version Log Format
| Date | Editor | File or Area | Summary of Change |
| --- | --- | --- | --- |

## Maintenance Rules
1. Update the relevant module README when adding a new file.
2. Update the root README when adding a new program-level asset.
3. Update the handoff-package indexes when navigation changes.
4. Keep print-ready HTML templates synchronized with their markdown source copy.
5. Preserve generic source materials when creating branded variants.

## Recommended Review Cycle
- Monthly: Check links, rollout docs, and signoff materials.
- Quarterly: Refresh examples, screenshots, and branding.
- After each training cohort: Capture facilitator and supervisor feedback, then revise weak materials.

## Change Control Note
If Georgia Mechanical creates additional client-branded variants, consider adding a separate client folder structure rather than overwriting shared source files.
